# Реверс-инжиниринг BK7252N — карта регистров камеры (high confidence)

Источник: стоковый дамп `CY365_Car_IP_EEE-249530-GDVHL_1.7.0` — **ровно та же камера**, что у заказчика
(подтверждено). Вход: UART-лог (`go os_addr(0x10000)` → app линкуется с **0x10000**), даташит
`DS-BK7252N-E01 V1.0.pdf`. Рабочие файлы: `stock_bk7252n/`.

## 0. Ключевой технический факт: формат флеша

Дамп — в формате Beken **CRC-interleave: 32 байта данных + 2 байта CRC** (с самого начала, шаг 34).
Из-за этого «сырой» дамп нельзя анализировать напрямую (каждые 32 байта смещение «съезжает» на 2).
Я **де-интерлейснул** образ → `stock_bk7252n/firmware_deinterleaved.bin` (1.97 МБ, логический образ).
Проверка успешна: строки стали целыми (`jpeg opened`, `Detected GC0311 [`), а 32-байтное окно таблицы
инициализации GC0311 из opencam **совпало** в де-интерлейснутом образе → init сенсора переиспользуем.

## 1. Подтверждено (высокая уверенность)

- **Ядро ARM968E-S** (валидная ARM-таблица векторов @0x0). То же, что у opencam.
- **RT-Thread 3.1.0**, Beken BDK (линейка `beken378`), BLE 5.2, прошивка `BK7252N_1.0.14`, сборка дек-2024.
- Конвейер камеры (по даташиту + коду): **сенсор → 8-bit DVP/CIS → аппаратный JPEG → выделенный DMA → RAM**.
- Сенсоры с автоопределением: **GC0311, GC0312, GC0328, HI704, OV7670, OV7673, SP0A38** — все нужны.
- Конфиг — EasyFlash + LittleFS во флеши (SD не нужна).

## 2. ГЛАВНОЕ: у BK7252N другая карта периферии

opencam (старый BK7251/7252) и BK7252N адресуют периферию **по-разному**. Камерная подсистема
BK7252N вынесена в **новый регион `0x00A0xxxx`, которого в opencam НЕТ вообще**.

| Подсистема | opencam (старый BK7252) | **BK7252N (из стока, путь B)** | Уверенность |
|---|---|---|---|
| **DVP / CIS захват** | в составе JPEG | **`0x00A04000`** (ctrl `0xA04008` бит0=en; геометрия `14/18/1C/34`; буфер `0xA04020/30`; trig `0xA04024`; IRQ `0xA04028`) | средне-высокая (декодирована функция настройки кадра) |
| **JPEG-энкодер** | `0x0080A000`/`0x00807000` | **`0x00A03000`** (reset/enable `0xA03034` бит `0x10`; cfg `0xA03018`) | средне-высокая (из `jpeg_open`) |
| **I2C/SCCB сенсора** | APB I2C `0x00806400` | **`0x00802410`** (ctrl `0x802410` бит5=en, addr `[14:8]`; status `0x802418`; rx `0x80241C`; trig `0x802480`; data `0x802484`) | средне-высокая (обработчик cmd `0x0E29xxxx`) |
| **Тактирование камеры** | SCTRL `0x00800000` | SCTRL **`0x00800180`** (ctrl: бит0=reset, бит3/6=clk-en; делители `184/188/198/19C`; 64-бит счётчик `18C+1A8`) | средне-высокая (прямые регистры) |
| **Выходной DMA** | GDMA `0x00809000` | GDMA `0x00809000` (тот же; каналы `+0x40`) | высокая |
| QSPI/SPI-флеш (НЕ камера) | `0x0080D000` | `0x00A02000` (опкоды `0xEB/0x02/0x20/0x9F`) | высокая |

> ⚠️ Поправка по пути: блок `0x00A02000` сначала ошибочно принял за DVP — при дизассемблировании
> оказался **контроллером флеша** (опкоды SPI-NOR). Реальный DVP — `0x00A04000`; I2C — `0x00802410`.
> Это показывает, почему атрибуцию надо подтверждать дизассемблированием функций, а не по диапазону.

**Карта камеры BK7252N восстановлена (путь B), детали — в `bk7252n-opencam-port/bk7252n_cam_regs.h`.**
Конвейер: сенсор ←I2C `0x802410`→ настройка; пиксели →DVP `0xA04000`→ JPEG `0xA03000`→ GDMA `0x809000`→ RAM;
тактирование/сброс — SCTRL `0x800180`. Остаётся уточнить отдельные битовые поля (формат/геометрия/quant) —
это делается на железе по UART-логу, либо сверкой с Beken SDK 3.0.76.

Полная MMIO-карта BK7252N (по частоте обращений): `0x00802000` (APB, x446), `0x0080D000`,
`0x00800000` (SCTRL/ICU), `0x00900000` (Wi-Fi MAC?), `0x00803000`, `0x00809000` (GDMA), `0x0080B000` (BLE),
`0x00A02000` (флеш), `0x00A03000`/`0x00A04000`/`0x00A06000` (камера/JPEG), `0x00C00000` (RAM/PSRAM?).

**Вывод (высокая уверенность, и он не зависит от точной атрибуции блоков):** opencam как есть
на BK7252N работать НЕ будет — он пишет JPEG по `0x80A000`/`0x807000`, которых на BK7252N **нет**
(реальная периферия камеры — в регионе `0x00A0xxxx`). Но порт **реалистичен**: регион найден,
JPEG-enable идентифицирован, а верхний уровень (init сенсора, схема DVP→JPEG→DMA) совпадает с opencam.

## 3. Что нужно для порта

1. **Драйвер камеры** переписать на адреса `0x00A02000/03000/04000/06000` (см. таблицу). Источник истины:
   либо Beken SDK 3.0.76+ (там этот драйвер уже есть — Маршрут A), либо доковырять битовые поля из стока
   по этим адресам (Маршрут B; адреса уже найдены).
2. **Мультисенсор**: оставить автоопределение всех 7 сенсоров (таблицы — из opencam/стока, чип-независимы).
3. **App-слой** (чип-независимый, пишется параллельно): HTTP-MJPEG `multipart/x-mixed-replace`,
   Wi-Fi STA + SoftAP fallback, выпил P2P/MQTT, конфиг в EasyFlash.
4. **Сеть**: SSID `opencambk`, пароль `imeXScdxADaUHuUU` (запас: `stream-lan-no84`).

## 4. Статус по маршрутам

- **A (чистый):** взять Beken BDK 3.0.76+ (есть драйвер камеры под `0xA0xxxx`) + пришить стример opencam.
  Мои находки дают возможность **проверить** любой найденный SDK на соответствие реальному железу.
- **B (по стоку):** продолжить RE битовых полей DVP/JPEG по найденным адресам — дольше, но без зависимости от SDK.

Личный дамп заказчика после снятия переделается быстро: процедура (де-интерлейв → поиск камерных строк →
карта регистров) уже автоматизирована в `scratchpad/*.py`.
